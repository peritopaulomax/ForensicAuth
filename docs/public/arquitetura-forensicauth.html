1. O que é o ForensicAuth

O ForensicAuth (Verificação de Autenticidade) é uma aplicação para peritos e analistas que precisam:

  • Organizar investigações em casos com número de protocolo;
  • Receber evidências (imagens, áudio, vídeo, PDF) preservando integridade (SHA-256);
  • Aplicar técnicas forenses (ELA, PRNU, metadados, estrutura PDF, áudio ENF, etc.);
  • Registrar apenas ações deliberadas em uma cadeia de custódia imutável;
  • Fechar casos com manifesto assinado e exportar/importar pacotes VCP entre instituições.
Princípio central: explorar parâmetros de análise (preview) não altera a cadeia. Só entram eventos como upload, derivado salvo, laudo, compartilhamento, fechamento e importação/exportação VCP.

3. Fluxo do perito (visão funcional)

  1. Login — JWT identifica quem executa cada ação.
  2. Criar caso — protocolo único, metadados processuais.
  3. Upload de evidência — arquivo gravado em disco; SHA-256 registrado; evento evidence_upload na cadeia.
  4. Análise exploratória — perito escolhe técnica e parâmetros; job roda no worker; resultados em results/ (sem custódia).
  5. Salvar derivado — artefato escolhido vira evidência derivada; evento derivative_saved com proveniência (insumos, algoritmo, parâmetros).
  6. Verificar integridade — “Verificar cadeia” (log) ou “Verificação forense” (log + arquivos + assinaturas).
  7. Compartilhar / fechar — colaboração auditada; fechamento com manifesto assinado.
  8. Exportar VCP — pacote ZIP portável para outra instalação ForensicAuth.

4. Backend e motor forense

4.1 Estrutura em camadas

  • api/v1/endpoints/ — rotas HTTP (casos, evidências, análise, auditoria, VCP).
  • services/ — lógica de negócio (custódia, jobs, exportação, lifecycle).
  • models/ — entidades SQLAlchemy (User, Case, Evidence, CustodyRecord, …).
  • core/plugins/ — adaptadores que implementam ForensicPlugin.
  • core/legacy/ — implementações algorítmicas preservadas (jpegio, PRNU, parsers binários, etc.).

Cada técnica (ex.: jpeg_ghosts, prnu, pdf_structure_metrics) é um plugin registrado no startup. O plugin valida parâmetros, executa o algoritmo e devolve artefatos (imagens, HTML Plotly, JSON).

4.2 Jobs de análise

Um AnalysisJob liga uma evidência a uma técnica e parâmetros. Estados: pending → running → completed | failed. Com PostgreSQL em produção, o worker Celery consome a fila Redis. Previews não geram registro de custódia; apenas derivados salvos pelo usuário.

5. Cadeia de custódia (resumo)

Cada evento forense relevante vira um registro append-only em custody_records:

  • record_hash = SHA-256 de payload canônico (tipo, IDs, hashes de arquivo, detalhes JSON, elo anterior, sequência).
  • previous_record_hash liga o registro ao anterior — cadeia por caso.
  • system_signature = Ed25519 sobre record_hash (identidade da instalação).

Alterar qualquer campo antigo quebra a verificação. Detalhes completos: documento Cadeia de custódia e verificação forense.

7. Transferência entre instalações (VCP)

O Verification Case Package (VCP) é um ZIP (.vcp.zip) com JSON canônico, binários endereçados por SHA-256, cadeia completa, fechamentos e chave pública Ed25519 da origem. Importação valida tudo antes de gravar; conflitos de protocolo/caso ativo são rejeitados.

Referência: Estrutura do Verification Case Package.

8. Implantação típica

Produção recomendada: Docker Compose no Linux com PostgreSQL, Redis, app, worker e frontend nginx. TLS no reverse proxy externo. Backup diário do banco + volumes de evidências + chaves Ed25519.
